# 针对我这份简历的面试题库

这份文档不是通用八股，而是围绕我当前简历里最容易被面试官追问的内容来设计。

目标：
- 提前准备“顺着简历就会问到”的问题
- 覆盖简单题、中等追问、困难深挖、压力测试题
- 区分“必须讲清楚”和“会被追问时再展开”的内容

建议使用方式：
1. 先把每个项目的“基础题”答顺。
2. 再把“深入题”里的方法论和 tradeoff 讲清楚。
3. 最后准备“困难题/压力题”，避免被连续追问时失守。

## 一、这份简历最容易被追问的主线

面试官大概率会顺着下面 5 条主线深挖：

### 1. 腾讯：视频 caption / omni caption / 多镜头 caption
- 为什么从纯视频演进到带音频、再到 omni caption
- 为什么要做 schema 演进
- 为什么是多阶段 pipeline，而不是一步生成
- 音画一致性怎么定义，怎么评估
- 你在这个系统里的 ownership 到底是什么

### 2. 腾讯：SFT / GRPO / 评测闭环
- SFT 和 GRPO 分别解决什么问题
- reward 怎么设计
- 评测集怎么做
- 为什么你们的优化是真提升，不是指标投机
- 80%+ 到底是什么指标

### 3. 腾讯：多模态预训练与数据
- 10 亿级图文数据怎么清洗
- CLIP 过滤是怎么做的
- 为什么数据比模型结构更关键
- VQA / OCR / Grounding 的训练数据和评测怎么定义

### 4. Shopee：推荐排序与多目标建模
- 为什么目标从 CTR 变成 CTR * CR
- 为什么 MLP 不够，要升级到 MMOE
- 多目标建模为什么会提升下单
- 线上实验怎么验证是因果有效

### 5. 小米：推荐、广告、冷启动
- LightGBM、ESMM、fastText 分别解决什么问题
- 标签挖掘怎么做
- 冷启动怎么设计
- 这段经历和多模态岗位有什么关系

## 二、面试官会怎么判断这份简历

这份简历的强项很明显：
- 多模态视频 caption 方向比较连续，不像临时转方向
- 不只是做模型，而是做数据、训练、评测闭环
- 有推荐系统和工业落地背景，说明不是纯研究

风险点也很明显：
- 面试官会怀疑你是不是“以数据和 pipeline 为主，模型深度不够”
- 面试官会追问你在 SFT / GRPO 中到底做了多少核心工作
- 面试官会追问那些指标是不是内部口径、是否足够扎实
- 面试官会测试你对 Transformer、MHA、KV cache、RoPE 这些基础是否真的很熟

所以策略是：
- 先把 ownership 讲硬，不要让人觉得你只是协作者
- 再把方法论讲清楚，证明不是工程堆砌
- 最后把基础算法题准备扎实，避免“项目很强但基础不稳”

## 补充：代码里能坐实的事实

下面这些点不是空泛推断，而是当前代码仓里能找到对应脚本、文档或配置的内容。面试里优先讲这些，会比抽象表述更有说服力。

### 1. 这不是单 prompt 项目，而是完整多阶段 pipeline
- `hunyuan_caption_pe` 的 README 明确把流程拆成了 Preprocess、Stage 1 Caption、Stage 2-4 QA Revision。
- preprocess 里包含 scene detect、qwen3-omni 语种检测、Seed-ASR、Qwen keyframe、prescan、audio caption、OCR。
- revise 不是一句话带过，而是 `qa_generate -> qa_answer -> revise` 三段式闭环。

### 2. 音频链路有非常具体的机制，不是“加了 ASR 就更好”
- 代码里是先做 qwen3-omni 语种检测，再用指定语言跑 Seed-ASR，最后把带时间戳的 ASR prior 注入 audio caption。
- 这套链路的目的不是单纯提高 ASR，而是减少字幕/OCR 对音频理解的污染。
- 这点很适合用来回答“为什么 prompt 修不掉音频幻觉”。

### 3. 评测体系不是一个总分，而是至少两套独立信号
- `llm-api/EVAL.md` 里有 QA 准确率评测：401 个视频、1349 道题。
- 同一个文档里还有时间戳 IoU：字幕 80 条、音频 75 条。
- 这说明你们不是只看生成文本像不像，而是同时看内容理解和时间定位。

### 3A. 现在统一 PE 的 benchmark 结果可以怎么讲
- 你最新给的表里，`pe, 0324` 在 `0319 QA` 上是 `78.36 / 12.89 / 8.73`。
- 同一个版本在 `0324 QA` 上准确率是 `77.68%`。
- 横向看 `0319 QA`，`qwen3omni (currently trained)` 是 `68.10%`，`Gemini 2.5 Flash 直出` 是 `63.60%`，`Gemini 2.5 Pro 直出` 是 `73.89%`。
- 更稳妥的外部说法不是“80%+”，而是“统一 PE 在两版 QA 集上稳定在 77%-78%，并显著优于当前训练版和直出方案”。

### 4. 训练数据侧有明确规模和过滤逻辑
- `llm-api/DATA.md` 写的是多镜头 SFT 数据约 20w。
- 来源是两批各 100k，从 360w 视频池按类目配额采样，且第二批去重不重叠。
- 数据不是裸 caption，而是结构化 JSON，包含 `visual_events`、`overlay_events`、`audio_events`、`audio_summary`。
- 过滤逻辑包括 shot 数限制、audio 结果注入、OCR/title/subtitle 注入、本地路径替换、ASR 替换对话文本。

### 5. RL/reward 也有可落地的实现，不只是概念
- `reward_functions/qa_rewards.py` 和 `timestamp_rewards_new.py` 说明 reward 至少分成 QA 和 timestamp 两类。
- 这两类 reward 都通过外部 vLLM server 调用 Qwen3-32B 做判断。
- QA reward 更偏 caption 是否支持题目答案，timestamp reward 更偏字幕/行为/对话定位是否对齐。
- 从训练脚本看，你们至少有两条线：`offline MPO` 和 `online GRPO`。
- `offline MPO` 脚本里是 `loss_type sigmoid bco_pair sft`，也就是 preference loss 和 SFT anchor 一起做联合优化。
- `online GRPO` 脚本里是单独的 `loss_type grpo`，重点是 rollout 后基于 reward 的相对优化。

### 6. 训练配置里有一些很能说明你“真看过训练脚本”的细节
- `train_core_ts.sh` 里能看到 Megatron SFT、`max_length=20480`、`fps=4`、`video_max_token_num=400`。
- 同时用了 `tensor_model_parallel_size=4`、`sequence_parallel=true`、`attention_backend=flash`。
- 这些可以用来支撑你对长序列训练、并行训练、FlashAttention 的理解。

### 7. 工程侧也能说明你们不是“能跑就行”
- `vllm_server/monitored_vllm.py` 里做了 health check、异常输出检测、自动重启。
- 这类细节可以用来回答“线上/离线大规模评测怎么保证稳定性”。

### 8. 有些回归数字可以帮助你把故事讲实
- `process.md` 里记录了一版 GT-50 回归：ASR 多语言 fallback 后，总体准确率从 43.67% 到 45.19%，speech 维度 +5.9%。
- 同一份日志还记录了 147 个 audio capsbench 样本的对比，以及多语言 fallback 命中的语种。
- 如果这些数字你确认可以对外说，就能很好地证明“音频链路优化是真有收益的”。

## 补充：不要主动说满的点

这些地方一旦说太满，面试官很容易顺着问到你兜不住。

### 1. 不要把 `80%+` 和 QA benchmark 准确率混为一谈
- `80%+` 更像内部生产链路下的核心可用性/准确率口径。
- `401 视频 / 1349 题` 的 QA 评测是 challenge bench，难度和口径都不同。
- 面试时最好主动先澄清：“这两个数字不是一个口径。”
- 现在你已经有统一 PE 的更干净结果了，所以外部面试里最好优先讲 `77%-78%` 这个 benchmark 口径，而不是死守 `80%+`。

### 2. 不要把 reward function 全说成自己独立设计
- 如果你没有从头写 trainer 和 reward 代码，就说自己主责的是 reward 目标、评测维度、case 分析和训练闭环配合。
- 这样更真实，也更安全。

### 3. 不要把 20w 数据集说成自己一个人构造完
- 更安全的说法是：你主导或深度参与了采样策略、字段设计、过滤逻辑和训练可用性闭环。

### 4. 不要把所有训练基础设施都说成自己搭的
- 你可以讲自己理解 `Megatron / Deepspeed / vLLM` 的训练和推理流程。
- 但如果没有直接 owning 整个框架，就不要说成“我从 0 搭了训练底座”。

### 5. 不要在外部面试里报过细的内部路径和机器信息
- 可以讲方法和规模。
- 不要主动讲 host、内网路径、具体机器 IP 这种实现细节。

### 6. 不要把“做数据和 pipeline”当成防守姿态
- 更强的说法是：你做的是多模态系统优化，数据、训练目标、评测和模型是一起设计的。
- 这样才不会被面试官轻易打成“只会拼链路”。

## 三、腾讯：视频 caption / omni caption / 多镜头 caption

这是最核心主线，必须准备到能连续讲 15 分钟。

### 开场时建议主动补的 4 个具体点

- pipeline 不是单阶段，而是 preprocess、多路信息融合的主 caption、以及 `qa_generate -> qa_answer -> revise` 的修订闭环。
- preprocess 里至少有镜头切分、语种检测、Seed-ASR、audio caption、Qwen keyframe、prescan、OCR 这些信号。
- 评测不是一个总分，而是 QA 准确率和时间戳 IoU 两套体系。
- 数据最终是结构化 JSON，不是普通自由文本，这也是你们能做训练、reward 和评测闭环的前提。

### A. 基础题

#### 1. 你最近最核心的项目是什么？

答题目标：
- 用 1 分钟讲清楚项目边界、演进和你的角色

答题要点：
- 这是视频 caption 方案持续迭代项目
- 从纯视频 caption 演进到带音频的多镜头 caption，再到统一 caption / omni caption
- 你不仅做模型，还做结构化表达设计、数据生产、训练和评测闭环
- 如果面试官愿意听细节，可以顺手补一句：主流程里有多路信号融合和 QA revise，不是单次生成

失分点：
- 只讲“做了 caption”
- 不说演进路径
- 不说自己的 ownership

#### 2. 你在这个项目里的具体职责是什么？

答题要点：
- 方案演进与路线选型
- schema 演进与结构化表达设计
- 数据生产链路和多阶段 pipeline
- SFT / GRPO 训练配合
- 评测集与自动评测闭环
- 如果要补实锤，可以说评测拆成 QA 准确率和时间戳 IoU 两块，训练数据侧也做了结构化字段和过滤逻辑

失分点：
- 职责描述太散
- 听起来像“参与了很多”，但没有主责

#### 3. 为什么要从纯视频 caption 演进到 omni caption？

答题要点：
- 纯视频阶段先解决视觉理解和基础描述
- 实际场景里音频是关键增量信息
- 多镜头场景涉及镜头切换、对白、环境音、字幕等多源异构信息
- 统一的 omni caption 更适合训练数据构造和下游 T2V 需求

#### 4. 什么叫音画一致 caption？

答题要点：
- 不是“画面写一遍、音频写一遍”
- 是视觉事件、音频事件、字幕文字、时间顺序互相对齐
- 训练数据可用性的关键不只是丰富，而是一致

#### 5. 为什么要做 schema 演进？

答题要点：
- 如果不同阶段的数据格式完全割裂，训练和评测会很碎
- 但如果强行要求所有阶段永远共用一套完全相同的 schema，也不现实
- 多镜头 schema 和当前统一 PE schema 服务的目标并不完全一样
- 关键不是一套 schema 永远不变，而是结构化表达要跟着任务目标演进

### B. 中等追问

#### 6. 为什么要做多阶段 pipeline，而不是一个大模型一步生成？

答题要点：
- 一步生成可控性差
- 错误来源不容易定位
- 中间结果无法复用，不利于批量生产
- 多阶段便于做缓存、回归、局部优化和失败兜底

建议回答结构：
1. 先承认一步生成链路最短。
2. 再说工业系统里稳定性、可解释性、可回归更重要。
3. 最后说多阶段不是堆模块，而是拆问题。

#### 7. schema 演进时，最关键的设计原则是什么？

答题要点：
- 既要能表达多模态信息，又不能太复杂导致生产成本爆炸
- 字段要同时服务数据生产、训练和评测
- 字段之间要尽量可验证，避免自由文本完全不可控
- 时间信息要能回到原视频核查
- 不同阶段可以有不同 schema，但演进逻辑必须清楚

可能追问：
- schema 太复杂会不会让模型更难学？
- 你们怎么处理缺失字段？
- 多镜头 schema 和统一 PE schema 的主要差异是什么？

#### 8. 多镜头 caption 比单镜头难在哪？

答题要点：
- 镜头切换导致时序和主体连续性更复杂
- 事件跨镜头对齐困难
- 音频和字幕可能跨镜头延续
- 容易出现“局部对，但整体时序错”

#### 9. 你说音频幻觉，具体是什么现象？

答题要点：
- 模型看到字幕/OCR，就误以为听到了对应内容
- 视觉文本反过来污染音频理解
- 这是系统性误差，不是个别 bad case

#### 10. 你们怎么发现这个问题不是 prompt 能解决的？

答题要点：
- 通过 case 分析发现错因稳定复现
- 不同 prompt 下只是表现形式变化，根因还在
- 说明问题不是“不会答”，而是输入信号优先级错了

### C. 困难深挖题

#### 11. ASR prior 注入为什么有效？

答题要点：
- 给音频链路引入可信锚点
- 降低视觉字幕对音频理解的污染
- 把模型从“自由猜”变成“围绕可信语音线索补全”
- 对多语言场景尤其重要

面试官真实目的：
- 测你是否能把一个工程经验抽象成机制解释

#### 12. 为什么不是直接端到端训一个更强的 omni 模型？

答题要点：
- 不是不能做，而是成本、数据质量、错误可解释性、生产稳定性都要一起考虑
- 如果结构化表达和评测口径都不稳定，直接端到端通常会把问题藏起来
- 工业系统更看重闭环可控，不是单次 benchmark 漂亮

#### 13. 你们怎么定义“音画一致性”？

答题要点：
- 视觉事件和音频事件是否对应
- 字幕/ASR 与最终描述是否冲突
- 事件时间顺序是否正确
- 主体、动作、说话内容是否一致

建议不要只说抽象概念，要说可评测维度。

#### 14. 你们怎么构建评测集？

答题要点：
- 先明确任务维度：内容理解、时间定位、音画一致、事实性
- 再选覆盖高频场景和 tricky case 的样本
- 要包含容易出错的 case，而不只是平均样本
- 自动评测和人工 case review 一起做

#### 15. 你怎么证明一个版本真的更好？

答题要点：
- 自动评测核心指标提升
- tricky case 集合不过回归
- 人工抽样可视化结果更稳
- 训练数据可用性提升，而不是只看生成文本更长
- 如果要给具体数字，当前统一 PE 可以直接讲：0319 QA 上 78.36%，高于 qwen3omni 当前训练版 68.10% 和 Gemini 2.5 Pro 直出 73.89%

### D. 压力测试题

#### 16. 听起来你更像是做 pipeline 和数据，不像做模型的人，你怎么看？

回答策略：
- 不要防御，不要否认数据和 pipeline 的重要性
- 直接把自己的定位讲清楚：做的是“多模态系统优化”，不是单点模型调用
- 强调自己参与了方案选型、SFT、GRPO、评测定义和数据闭环

可用回答：

我的工作确实不只是在调单个模型，我更偏向把多模态 caption 这件事做成一个可持续优化的系统。但这不意味着模型部分弱，恰恰相反，结构化表达、数据、训练目标和评测口径没有设计好，模型优化很容易失真。我做的是把模型能力和数据闭环真正对齐，而不是只追单点实验。

#### 17. 你简历里写了很多“负责”，哪些是你亲自做的，哪些是团队一起做的？

回答策略：
- 明确区分 ownership 和团队协作
- 不要把所有事都说成自己独立完成
- 但要把你主导的关键决策说实

#### 18. 如果让你现在从 0 到 1 重做这套系统，你会改哪三件事？

答题要点：
- 更早明确 schema 演进路径
- 更早建设 tricky case 和自动回归
- 更早明确音频链路的可信先验

## 四、腾讯：SFT / GRPO / 评测闭环

这部分最容易暴露“是不是只会说概念，不知道训练细节”。

### A. 基础题

#### 1. SFT 和 GRPO 各自解决什么问题？

答题要点：
- SFT 解决基础能力、格式稳定性、描述完整性
- GRPO 解决事实性、时间定位、音画一致性等更细的对齐问题
- 先把会说学出来，再优化说得对、说得稳

#### 2. 为什么 SFT 不够，还要 GRPO？

答题要点：
- SFT 本质上是行为模仿
- 对局部幻觉、排序错误、可验证约束不够稳
- GRPO 可以围绕 reward 强化关键维度

#### 3. reward 大概怎么设计？

答题要点：
- 以可验证维度为主
- 围绕 QA、时间定位、事实一致性设计
- reward 必须和真实目标相关，且能稳定自动打分
- QA reward 一个很关键的点是，同一次 rollout 里要有区分度，不能长期全 0 或全 1
- 如果 reward 没有区分度，GRPO 很难学到“哪种回答更好”

#### 3A. 如果面试官问 loss 怎么写，你可以怎么答？

答题要点：
- 先区分 `offline MPO` 和 `online GRPO` 两条线
- `offline MPO` 更像联合目标：`L = w1 * L_pref + w2 * L_pair + w3 * L_sft`
- 当前脚本里对应的是 `sigmoid + bco_pair + sft`
- `L_sft` 的作用是 anchor，避免偏好优化把基础生成能力带偏
- trainer 实现里是按 `loss_type` 列表逐项算、按 `loss_weights` 加权求和
- `bco_pair` 还有单独实现，不是完全复用默认 DPO loss
- `online GRPO` 则是基于 reward 的相对优化，不要和 MPO 的联合 loss 混为一谈

### B. 中等追问

#### 4. 为什么用 GRPO，而不是 PPO / DPO？

答题要点：
- 不要空讲定义，要结合任务说
- 重点讲你们场景里为什么需要基于规则或可验证 reward 的优化
- 如果没有很深的算法细节 ownership，就把重点放在“为什么这个框架适合 caption 对齐”

安全说法：
- 我们重点关注的是让模型在可验证的 caption 质量维度上优化，因此比起纯偏好式方法，更需要围绕规则和评测信号构建 reward。具体训练框架上，核心不是名字，而是 reward 是否稳定、训练是否可控。

#### 4A. 为什么联合 loss 里要保留 SFT 项？

答题要点：
- 纯偏好优化容易把基础 caption 能力带偏
- SFT 项相当于 anchor
- 它能帮助模型保留格式稳定性、描述覆盖率和语言质量
- 所以联合 loss 的重点不是“多加一项”，而是控制偏好优化的副作用

#### 5. 你们怎么避免 reward hacking？

答题要点：
- reward 不做单一指标
- 加人工 review 和 tricky case 回归
- 看 reward 提升是否伴随真实可视化质量提升
- 防止模型学会投机输出模板
- 还要避免 reward 退化成没有区分度的信号，比如一批 rollout 全 0 或全 1

#### 6. 你们怎么判断 RL 阶段没有把基础生成能力搞坏？

答题要点：
- 保留基础评测集
- 看结构化输出是否退化
- 看描述覆盖率是否下降
- 看典型 case 是否出现新型错误

### C. 困难深挖题

#### 7. 你们的 reward 如果包含多个维度，怎么平衡？

答题要点：
- 不同维度权重需要实验调优
- 一般先保证硬约束维度，再调细节维度
- 不希望模型为了某一维度极致优化而牺牲整体可用性

#### 8. 你们有没有遇到训练不稳定、模式坍缩或者格式漂移？

答题要点：
- 这种问题大概率出现过
- 要讲排查路径：数据、reward、采样、训练超参、格式约束
- reward 侧一个典型检查项就是：同一次 rollout 的答案分数有没有拉开，是否退化成全 0 / 全 1
- 如果是联合 loss 训练，还要看 SFT anchor 是否太弱，导致模型被偏好项拉偏
- 即便没有深入到 trainer 实现，也要会讲定位方法

#### 9. 评测闭环为什么比单次提分更重要？

答题要点：
- caption 是持续迭代任务
- 不同版本之间容易发生局部回退
- 没有回归体系，线上长期效果不稳

### D. 压力测试题

#### 10. 你说参与了 SFT 和 GRPO，具体到训练环节你做了哪些事情？

回答策略：
- 不要冒充自己写了全部 trainer
- 要说清你真正做的部分：数据构造、训练目标设计配合、评测、case 分析、反馈闭环
- 如果也参与了配置、样本构造、reward 定义，就说具体

#### 11. 80%+ 指标到底是什么？为什么听起来有点虚？

回答策略：
- 承认这是内部评测体系指标
- 说明覆盖维度和评测流程
- 强调这是稳定口径下的版本对比结果，不是单次 demo
- 如果对方明显更关心 benchmark，就直接切换成统一 PE 当前结果：`0319 QA 78.36%`、`0324 QA 77.68%`

## 九、追问型问题

这部分不是基础题，而是面试官已经听完你的项目介绍后，开始判断你到底是“懂模型的人”，还是“只会讲流程的人”时最容易问的追问题。

### 76. 听起来你更像做数据和 pipeline，不像做模型的人，你怎么看？

参考回答：

我理解面试官为什么会有这个感觉，因为我的项目里确实有很多数据、schema、评测和生产链路的工作。但我的理解是，多模态项目里真正有价值的模型工作，很多时候不是只改一层网络结构，而是把训练目标、数据表达、评测口径和优化方式打通。

我做的不只是数据搬运，而是定义什么样的数据表示更适合训练，什么样的评测信号能真正反馈模型质量，以及 SFT、MPO、GRPO 这些训练环节怎么和任务目标对齐。像音频幻觉、QA reward 区分度、schema 演进这些问题，本质上都不是纯工程问题，而是直接影响模型学什么、怎么学的问题。

所以如果要定位，我会说我是偏训练闭环和多模态建模落地，而不是只做模型底座结构的人。

更口语一点的版本：

我不太把模型工作理解成只改 backbone 或者只写 loss。对多模态项目来说，模型到底学到什么，很多时候是由数据表达、训练目标和评测信号一起决定的。我做的工作虽然看起来有很多 pipeline 和数据，但其实核心是在把任务目标转成模型能学、评测能验、系统能迭代的闭环。

### 77. 你说参与了训练，具体参与到哪一层？

参考回答：

我参与的主要不是底层框架算子或者 base model 预训练，而是任务层训练闭环。具体包括几块。

第一块是训练数据构造，包括多镜头和统一 PE 数据怎么筛、怎么组织、怎么定义输入输出结构。

第二块是训练任务设计，比如 SFT 阶段要优化什么，RL 阶段 reward 怎么定义，什么样的样本更适合进 MPO 或 GRPO。

第三块是评测和回流，也就是训练之后怎么判断模型是真的变好了，还是只是某个表面指标涨了。

如果往更细说，我会把自己的工作定位在“任务定义、训练数据、reward 和评测闭环”这一层，而不是底层 CUDA 或模型主干结构魔改。

更口语一点的版本：

如果按层次分，我更偏任务层训练，而不是底层训练框架。我主要参与的是数据怎么组织、目标怎么定义、reward 怎么配、评测怎么闭环，以及训练后怎么做 case 分析和回流，不是去改最底层的算子或者从零写 trainer。

### 78. 你说做了 schema，具体字段和原则是什么？

参考回答：

schema 我不想说成一个固定模板，因为多镜头阶段和统一 PE 阶段其实不是完全同一套。

如果讲核心字段，大致会包括几类：

- 视频级字段：`style`、`summary`、全局背景音、整体场景信息
- 镜头级字段：每个 shot 的起止时间、镜头内容描述、关键动作、人物、场景切换
- 音频相关字段：台词、说话人特征、音效、背景音乐、环境音
- 对齐字段：时间戳、shot boundaries、音画对应关系
- 结构化输出字段：便于 QA revise、评测和后处理的中间表示

我做 schema 时主要遵循几个原则：

- 第一，字段要服务训练目标，不是为了结构化而结构化
- 第二，字段要可评测，不然后面很难闭环
- 第三，字段要能兼容多源信息，包括视觉、字幕、音频和时间轴
- 第四，字段要能演进，因为多镜头阶段和统一 PE 阶段服务的任务不完全一样

所以 schema 在我这里不是“写个 JSON 格式”，而是训练目标和数据表达之间的接口设计。

更口语一点的版本：

schema 对我来说不是一个固定模板，而是数据表达和训练目标之间的接口。核心字段一般会覆盖视频级摘要、镜头级描述、音频事件、时间戳对齐和一些便于 revise、评测的中间结构。更重要的是字段设计原则，要能服务训练、能做评测、能接多源信息，而且要允许演进。

### 79. 你说 reward 有区分度问题，怎么发现的？

参考回答：

这个不是先从理论上发现的，而是从训练现象和 rollout 结果里发现的。

一开始我们会看同一个 prompt 下 rollout 出来的多个候选答案，如果 QA reward 经常出现一批候选基本全 0 或者全 1，那就说明 reward 虽然定义出来了，但对当前样本没有足够区分力。

这种情况下，训练表面上在跑，实际上相对偏好信号很弱，模型很难学到“哪个答案更好”。

所以我们后来不是只盯 reward 公式本身，而是会看 rollout 后 reward 的分布、不同候选之间能不能拉开差异、这种差异是否稳定。

这个过程本质上是从训练信号信息量的角度去检查 reward，而不是只看 reward 有没有定义。

更口语一点的版本：

我们不是先拍脑袋觉得 reward 有问题，而是先看 rollout 结果。后来发现有些题一轮 rollout 下来几个答案分数几乎一样，要么全对要么全错。这样从形式上看 reward 是有的，但实际上信息量很低，模型学不到稳定偏好。所以后面我们会专门看 reward 分布是不是能把候选拉开。

### 80. 你说 PE 比直出强，为什么不直接把 PE 蒸馏进模型？

参考回答：

可以做，但不是一句“蒸馏一下”就能自然吃进去，主要有三个现实问题。

第一，PE 的优势里有一部分来自更复杂的 pipeline、结构化约束和 revise 机制，这些能力并不一定能被一次性蒸馏进单步生成模型。

第二，PE 输出通常更结构化、更受控，但训练模型时如果数据构造和目标定义没对齐，蒸馏后很容易只学到表面格式，学不到真正的推理过程。

第三，蒸馏是值得做的，但前提是 schema、评测和 teacher 输出稳定，不然很容易把 teacher 的噪声也一起学进去。

所以我会把它看成下一阶段工作：先用 PE 验证什么样的表达和 revise 机制更有效，再把稳定部分沉淀成训练数据或 preference 信号，逐步蒸馏进模型，而不是一开始就假设单模型一定能完整替代 pipeline。

更口语一点的版本：

我不反对蒸馏，但 PE 强的地方不只是答案本身，还包括中间结构、约束和 revise 过程。这个不是把 final output 扔给模型学一遍就能完全吃进去的。更现实的做法是先把 PE 里稳定有效的部分沉淀成训练信号，再逐步往模型里内化。

### 81. 如果你来做下一版，你最优先改哪三件事？

参考回答：

如果是我来排优先级，我会先做三件事。

第一，继续提升训练信号质量。重点是 reward 区分度、样本难度分层，以及评测口径和训练目标的一致性。因为如果训练信号不够干净，模型优化很容易跑偏。

第二，继续优化音频链路。尤其是语音、音效、背景音乐之间的区分，以及音画时间对齐。音频这块目前仍然是有声视频 caption 最容易出错、但也最有增益空间的地方。

第三，推动 PE 能力向模型内化。不是一次性取代 PE，而是把已经验证有效的结构化输出、revise 信号和 teacher 结果，逐步沉淀到 SFT 或 preference 数据里，让模型本身能力更强，减少对外部 pipeline 的依赖。

更口语一点的版本：

如果我来做下一版，我会优先抓三件事：先把 reward 和评测信号做得更干净，再把音频链路继续补强，最后推动一部分 PE 能力往模型里内化。因为我觉得现在最大的空间还是训练信号质量、音频理解，以及单模型能力和外部 pipeline 的平衡。

### 82. 你从推荐转多模态，核心迁移能力是什么？

参考回答：

我觉得核心迁移能力有三点。

第一，问题拆解能力。推荐系统里经常要把目标拆成召回、排序、样本、标签、评测，多模态项目其实也一样，只是对象从用户行为变成了视觉、音频和文本。

第二，数据和目标对齐的意识。无论是推荐还是多模态，真正难的都不是“训个模型”，而是模型到底在学什么，训练目标和业务目标有没有对齐。

第三，闭环思维。推荐里我比较熟悉线上线下口径、指标回流、ab 迭代；到多模态这边，我会自然去关注 schema、训练、评测和 badcase 回流是不是闭环。

所以我不是把过去经验清零重来，而是把“目标定义、数据构造、评测闭环”这套能力迁到了多模态。

更口语一点的版本：

我觉得我迁过来的不是某个具体模型，而是三种能力：一是复杂问题拆解，二是数据和目标对齐，三是闭环迭代。推荐系统里这些能力很重要，到了多模态这边其实一样成立，只是输入从行为数据变成了视频、音频和文本。

### 83. 你最大的短板是什么？

参考回答：

如果实话实说，我最大的短板不是做事推进或者业务理解，而是在更底层的模型理论和框架细节上还不够深。比如强化学习理论、一些底层训练实现、模型主干结构演进，我能理解任务层和训练层的设计，但还不是那种专门做底座研究的人。

不过我对这个短板是清楚的，所以我最近会有意识地把 PPO、DPO、GRPO、attention、vLLM 这些基础问题补扎实。

从工作角度讲，我的优势还是在于能把问题落到可训练、可评测、可迭代的闭环里，这也是我现在最稳定的价值点。

更口语一点的版本：

如果说短板，我觉得自己在更底层的理论和框架实现上还可以再深，比如 RL 理论或者底座结构演进这些。但我不是完全没接触，而是深度还不如我在任务建模、训练闭环和评测这一层来得强。所以我现在也在有意识补这一块。

## 十、你可以反问面试官的问题

这部分不是凑流程，而是你可以借反问判断三件事：

- 这个岗位到底更需要什么能力
- 团队是做真核心问题，还是偏 support
- 你后面该把面试重心放在模型、训练还是落地上

原则：

- 反问不要太泛，不要像背模板
- 不要连续问福利、汇报线、加班强度这种非技术问题
- 最好围绕岗位目标、技术挑战、团队分工和成功标准来问

### 1. 这个岗位当前最核心的技术挑战是什么？

为什么值得问：

这个问题能快速看出团队现在到底卡在哪里，是模型能力、数据质量、训练效率，还是线上落地和产品指标。

更成熟的问法：

如果从未来半年到一年看，团队当前最核心的技术挑战是什么？更偏模型能力本身，还是数据、训练、评测和落地这一侧？

你要听什么：

- 如果对方回答很具体，说明团队目标清晰
- 如果只说一些空话，比如“都很重要”，说明方向可能还比较散

### 2. 团队更偏模型、数据、训练还是系统落地？

为什么值得问：

这能帮助你判断岗位是不是和你的优势真正匹配，也能判断面试官刚才为什么重点问你某些内容。

更成熟的问法：

我想再了解一下，这个岗位在团队里的定位更偏模型能力建设、训练和数据闭环，还是偏系统化落地和产品迭代？

你要听什么：

- 如果回答偏模型和训练，你后面就要强调训练、reward、结构理解
- 如果回答偏落地和系统，你的闭环能力反而会更加分

### 3. 这个岗位对候选人的核心期待是单点突破，还是闭环推进能力？

为什么值得问：

这个问题能帮你确认：他们到底想招一个专项很深的人，还是一个能把复杂系统往前推的人。

更成熟的问法：

如果站在这个岗位的视角，团队对候选人的核心期待更偏某个方向上的深入突破，还是更看重把一个问题从数据、训练、评测到落地整体推进的能力？

你要听什么：

- 如果对方强调单点突破，说明他们更偏研究或专项专家
- 如果强调闭环推进，说明你的背景会更匹配

### 4. 团队现在在多模态 / AIGC / 搜索里的主要方向是什么？

为什么值得问：

这是为了判断团队的主战场，而不是只看 JD 写了很多方向。

更成熟的问法：

我也想更具体了解一下，团队现在在多模态、AIGC、搜索这些方向里，当前最主要的投入重点是什么？是偏基础模型能力，还是更偏具体场景和产品化落地？

你要听什么：

- 听他们说的是“技术方向”，还是“业务方向”
- 听他们有没有一个明确主线，而不是很多方向都沾一点

### 5. 这个岗位一年后最希望看到我解决什么问题？

为什么值得问：

这个问题非常有价值，因为它会逼面试官给出更明确的成功标准。

更成熟的问法：

如果这个岗位招到合适的人，一年之后团队最希望他解决的是什么问题？更像是某个具体技术难题，还是把一个方向真正做出结果？

你要听什么：

- 如果回答具体，说明这个岗位目标很清楚
- 如果回答特别模糊，可能说明岗位边界还不稳定

### 反问时的使用建议

- 不要 5 个全问，通常挑 2 到 3 个最合适的就够了
- 如果对面是技术面试官，优先问技术挑战、岗位定位、成功标准
- 如果对面是 leader，再补团队方向和一年目标
- 如果前面聊得已经很具体，就不要重复问已经回答过的问题

一个比较自然的组合是：

1. 这个岗位当前最核心的技术挑战是什么？
2. 这个岗位在团队里更偏模型、训练，还是闭环落地？
3. 如果一年后回看，团队最希望我解决的是什么问题？

答题要点：
- VQA 更强调视觉理解与语言回答
- OCR 更强调细粒度文字识别和文本忠实
- Grounding 更强调位置对齐和区域定位

### C. 困难深挖题

#### 7. 如果 CLIP 过滤阈值设太高，会有什么问题？

答题要点：
- 数据量掉太多
- 长尾样本和复杂语义样本可能被误杀
- 训练分布会变窄

#### 8. 预训练数据和下游 SFT 数据的关系是什么？

答题要点：
- 预训练解决通用表示和对齐基础
- SFT 解决任务格式、任务能力和输出约束
- 两者不是替代关系

#### 9. 你觉得多模态模型效果提升更依赖模型结构，还是数据？

答题要点：
- 不要极端回答
- 但结合你的经历，通常可以强调数据和任务定义的上限更大
- 结构重要，但脏数据和错目标会让结构优势发挥不出来

### D. 压力测试题

#### 10. 你这段经历听起来像数据工程，模型创新在哪里？

回答策略：
- 强调工业多模态里“把训练数据和任务目标对齐”本身就是关键能力
- 不是只讲清洗，而是讲任务定义、数据选择、训练目标和评测联动

## 六、Shopee：推荐排序 / CTR * CR / MMOE

这一段会被用来测试你的推荐基本功，以及你是否真的做过线上优化。

### A. 基础题

#### 1. 为什么目标是 CTR * CR，而不是只优化 CTR？

答题要点：
- 点击不等于下单
- 只看 CTR 容易把流量推向标题党或低成交商品
- CTR * CR 更接近商业目标

#### 2. 为什么从 MLP 升级到 MMOE？

答题要点：
- CTR 和 CR 相关但不完全相同
- 共享底层表示有价值，但任务间也需要差异化建模
- MMOE 通过多个 expert 和 gate 平衡共享与个性化

#### 3. 多目标建模为什么会提升订单？

答题要点：
- 它不是只追求容易点的商品
- 兼顾点击意愿和转化概率
- 排序目标更接近真实业务收益

### B. 中等追问

#### 4. MMOE 相比 shared-bottom，核心优势是什么？

答题要点：
- 共享更柔性
- 不同任务可以从不同 expert 里取不同组合
- 缓解任务冲突

#### 5. 线上实验怎么验证结果可信？

答题要点：
- A/B test
- 看核心指标，也看副作用指标
- 观察周期要覆盖波动
- 分流随机性和样本量要足够

#### 6. 如果 CTR 涨了但 CR 跌了，怎么分析？

答题要点：
- 看样本结构变化
- 看是否流量偏向低质量高点击商品
- 看模型是否过拟合点击行为

### C. 困难深挖题

#### 7. 多目标训练里常见的问题是什么？

答题要点：
- 梯度冲突
- label 稀疏度不同
- 收敛速度不一致
- 某一目标主导训练

#### 8. 你怎么理解“目标设计比模型升级更重要”？

答题要点：
- 如果优化目标错了，模型再强也在优化错误方向
- 这和你现在做 caption 很一致，都是先定义好什么叫“好”

### D. 压力测试题

#### 9. 你从推荐转到多模态，核心迁移能力是什么？

回答策略：
- 强调不是换赛道，而是能力迁移
- 共同点是：目标定义、数据分布、评测体系、线上闭环

可用回答：

我虽然从推荐做到多模态，但本质上一直在做“复杂系统下的目标优化”。推荐里是排序目标、特征和实验闭环；多模态里是结构化表达、训练数据、评测和对齐闭环。两者底层方法论很一致，都是先把目标定义对，再围绕数据和模型做优化。

## 七、小米：广告 / 冷启动 / 标签挖掘

这部分一般不会问得像腾讯那样深，但可能作为补充验证你的工业经验。

### 1. LightGBM、ESMM、fastText 分别适合什么问题？

答题要点：
- LightGBM：结构化特征、效率高、工业常用
- ESMM：CTR/CVR 联合建模，缓解样本选择偏差

### 2. 冷启动场景为什么难？

答题要点：

- 统计特征不稳定
- 需要内容特征、标签特征或迁移信息补足

### 3. 标签挖掘的价值是什么？

答题要点：
- 提供更稳定的语义表示
- 服务召回、排序、人群理解
- 对冷启动和长尾都有帮助

### 4. 这段经历和现在找的大模型岗位有什么关系？

答题要点：
- 强调工业建模和效果优化经验
- 强调对数据噪声、目标定义、闭环迭代的理解
- 不要把它讲成“无关经历”

## 八、通用大模型基础题：你这份简历一定会被问

因为你写了多模态 LLM、Deepspeed、Megatron、vLLM，所以这些基础题必须能答。

### 1. MHA 的计算流程是什么？

必须能答：
- `Q/K/V` 线性映射
- split heads
- `QK^T / sqrt(d)`
- mask
- softmax
- 对 `V` 加权
- merge heads + output projection

### 2. 为什么 attention score 要除以 `sqrt(d)`？

必须能答：
- 点积方差随维度变大
- softmax 容易饱和
- 梯度会变差

### 3. KV cache 是什么，为什么能加速推理？

必须能答：
- 历史 token 的 K/V 可复用
- decode 时不需要重算全部历史
- 用显存换延迟

### 4. MHA、MQA、GQA 的区别是什么？

必须能答：
- 区别在 K/V 头数
- 核心 tradeoff 是效果和 KV cache 成本

### 5. RoPE 解决什么问题？

必须能答：
- 注入位置信息
- 在 Q/K 空间做旋转
- 让 attention 更自然建模相对位置

### 6. FlashAttention 为什么更快？

必须能答：
- attention 瓶颈常是 IO
- 避免显式物化完整 attention matrix
- 分块 + online softmax 降低访存

### 7. prefill 和 decode 的区别是什么？

必须能答：
- prefill 是并行处理整段输入
- decode 是逐 token 自回归生成
- 两者瓶颈不一样

### 8. Pre-LN 和 Post-LN 的区别是什么？

必须能答：
- Pre-LN 训练更稳定
- Post-LN 更接近原始 Transformer，但深层训练更难

## 十四、准备原则

最后提醒自己三件事：

### 1. 不要把所有工作都说成自己独立完成
这样容易被问穿。

面试官真正想看的是：
- 你怎么定义问题
- 你怎么拆问题
- 你怎么验证改进
- 你怎么形成闭环

如果这四点讲得清楚，这份简历是有竞争力的。
