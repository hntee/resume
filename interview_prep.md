# 面试问答稿

## 先统一口径

这几件事要先在自己脑子里分清，否则面试时很容易说混。

- 简历里的 `80%+` 不要直接等同于 benchmark QA accuracy。更稳妥的说法是：`80%+` 是内部生产链路下的核心可用性/准确率口径，而 `401 视频 / 1349 题` 是更细、更难的 QA benchmark。
- 多镜头评测至少有两条线：QA 准确率，以及时间戳 IoU。后者又分字幕 80 条和音频 75 条。
- 多镜头 SFT 数据不是“随便抓点视频生成 caption”，而是两批各 100k、从 360w 视频池采样后再做 audio / overlay / OCR / ASR 替换和过滤，最终约 20w。
- 现在这条 caption pipeline 可以讲成：preprocess 多源提纯 -> 主 caption 统一表达 -> QA revise 和评测闭环。
- 当前统一 PE 更适合对外讲的 benchmark 口径是：在两版 QA 集上准确率稳定在 `77%-78%`；在 `0319 QA` 上，统一 PE `78.36%`，高于 `qwen3omni (68.10%)` 和 `Gemini 2.5 Pro 直出 (73.89%)`。

## 从易到难

### 1. 你先简单介绍一下自己

参考回答：

我这几年主要在腾讯做视频理解和多模态 caption 相关工作，核心方向是视频 caption、带音频的 omni caption，以及多镜头 caption 的方案研发。工作内容不是只做单点模型，而是从 caption 格式设计、数据构造、训练到评测闭环一起推进。更早之前我在 Shopee 和小米做推荐和广告算法，所以也有比较完整的工业落地和效果迭代经验。现在我主要希望找偏多模态 LLM 落地、视频理解和数据训练闭环相关的岗位。

### 2. 你最近最核心的项目是什么

参考回答：

最近最核心的是 Hunyuan Video 训练数据的 caption / omni caption / 多镜头 caption 方案研发。这个项目经历了几个阶段：最开始是纯视频、没有音频的 caption；后来扩展到带音频的多镜头 caption；再往后收敛成统一的 caption / omni caption 方案。我的工作重点不是只做一个模型，而是把这条链路做成可持续迭代的系统，包括 preprocess 多源信息提纯、从多镜头 schema 到统一 PE schema 的演进、主 caption 生成、QA revise、SFT / GRPO 配合，以及评测闭环。

### 3. 你在这个项目里主要负责什么

参考回答：

我主要负责四块。第一是方案演进和选型，比如不同阶段怎么在纯视觉、带音频、多镜头和 omni 之间取舍。第二是结构化表达的设计与演进，也就是从早期多镜头 schema 逐步演进到当前统一 PE schema，让结果同时适合数据生产、训练和评测。第三是把 preprocess、主 caption 和 `qa_generate -> qa_answer -> revise` 这条链路打通。第四是训练和效果闭环，包括 SFT / GRPO 配合、评测集建设，以及围绕 QA 准确率和时间定位的自动评测优化。

### 4. 为什么要从纯视频 caption 逐步演进到 omni caption

参考回答：

一开始做纯视频 caption，主要是因为路径最短，先解决画面理解和基础描述能力。但后面很快发现，很多视频任务里音频是关键增量信息，比如对白、旁白、环境音、歌词，都会直接影响事实性和完整性。尤其到了多镜头场景，镜头切换、语音、字幕和画面之间的关系更复杂，如果只看画面会丢掉很多关键信息，所以后面才逐步演进到统一的 omni caption 方案。

### 5. 你说的“音画一致 caption”具体是什么意思

参考回答：

我理解的音画一致，不是简单把画面和音频都描述一遍，而是让最终 caption 在视觉事件、音频事件、字幕文字和时间顺序上保持一致。比如画面里是谁在做什么、对应的台词是谁说的、字幕和语音是否一致、事件先后顺序是否正确，这些都要能对齐。否则模型虽然生成了很多内容，但不一定适合拿来训练。

### 6. 你们为什么要持续演进 schema，而不是固定一版格式？

参考回答：

因为 schema 本质上是服务任务目标的，不是越固定越好。早期多镜头 schema 更强调镜头级事件、音频、字幕和时间轴的可评测性，便于做 QA 和 IoU 这类评测；后来的统一 PE schema 更强调多源信息融合、主 caption 生成、QA revise，以及最终训练数据的可用性。所以我做的不是从头到尾守着一版 schema 不变，而是根据训练目标、评测方式和生产链路去持续演进结构化表达。这个演进的核心价值，是让不同阶段的数据都能被稳定生产、训练和评测，而不是强行把所有场景压成完全相同的字段。

### 7. 你们为什么要做多阶段 pipeline

参考回答：

多阶段 pipeline 主要是为了把复杂问题拆开。直接让一个模型端到端做所有事情，虽然看起来简单，但可解释性、稳定性和可控性都差。拆成预处理、主 caption、QA 修订这些阶段后，不同信息源可以独立优化，失败样例也更容易定位问题。同时这样更适合做缓存复用和批量生产，工程上更稳。

### 8. 你提到 QA 自修正流程，这个是怎么工作的

参考回答：

核心思路是让模型先基于 caption 自己提出高价值验证问题，然后再基于视频回答这些问题，最后把回答结果和原 caption 做对比，只修正真正冲突的部分。这样做的价值是把“生成”变成“生成 + 核验 + 修订”的闭环。对 caption 这种任务来说，很多问题不是不会写，而是容易出现局部幻觉、时间顺序错误或者细节遗漏，QA 自修正对这些问题比较有效。

### 9. 训练数据是怎么构造的

参考回答：

训练数据不是单纯把模型输出直接拿来训，而是有一整套生产链路。以多镜头 SFT 数据为例，我们最终有一版大约 20w 的结构化数据，来自两批各 100k 的样本，从 360w 视频池按类目配额采样后去重合并。数据里不是只有 caption 文本，而是结构化的 `visual_events`、`overlay_events`、`audio_events`、`audio_summary`，同时还做了 audio/overlay 合并、OCR 注入、title/subtitle 过滤、本地路径替换，以及用 ASR 替换对话文本。我主要做的是把这些环节串起来，并保证它们能同时服务 SFT、RL 和评测。

### 10. SFT 和 GRPO 各自解决什么问题

参考回答：

SFT 主要解决的是“模型能不能稳定按我们想要的结构和粒度生成 caption”。比如输出是不是稳定遵循结构化 JSON，`visual_events`、`overlay_events`、`audio_events` 这些字段是不是完整、可用。GRPO 解决的是进一步对齐问题，比如事实性、时间定位、时序一致性和音画对齐。这些能力单靠 SFT 往往不够稳，所以后面会结合 reward 去优化，让模型不只是会生成，还要在 QA 和时间定位这类可验证维度上更可靠。

### 11. 你们的 reward 大概怎么设计

参考回答：

总体上是围绕结果可验证的维度设计。比较典型的两类信号，一类是 QA 相关的内容理解信号，另一类是时间定位信号。我们内部做法更偏向让 reward 对应自动评测能稳定衡量的东西，比如 caption 是否支持某道题的答案、某段字幕或对话的时间戳是否对齐。这里有一个比较关键的 know-how：以 QA reward 为例，不能让一次 rollout 里的样本长期退化成全 0 或全 1，那样相当于没有区分度，训练信号会塌。更有效的做法是让同一次 rollout 里不同答案能被 reward 拉开，有明显区分，这样 GRPO 才真的知道什么更好、什么更差。原则不是把 reward 设计得很花，而是让模型输出能被稳定自动打分，而且这个分数既和真实目标相关，又能提供足够的区分信号。这样 RL 才有意义，否则只是在优化一个和目标脱节、甚至没有梯度信息的代理指标。

### 11A. 你们 RL 里的 loss 是怎么写的？

参考回答：

这个我会先区分两条线来说，不把 MPO 和在线 GRPO 混在一起。

如果说的是我们那条 MPO / offline RL 线，从训练脚本上看，本质是把偏好学习项和 SFT 项一起做联合优化。脚本里是 `loss_type sigmoid bco_pair sft`，对应可以理解成总 loss 是一个加权和：

`L = w1 * L_pref + w2 * L_pair + w3 * L_sft`

其中：
- `L_pref` 可以理解成 DPO / sigmoid 这一类偏好项，核心是拉开 chosen 和 rejected 的相对偏好；
- `L_pair` 是 `bco_pair` 这类 pairwise 校正项；
- `L_sft` 是 teacher-forcing 的交叉熵项，作用更像 anchor，防止模型在偏好优化时把基础生成能力带偏。

从当前脚本看，权重是 `1.0 / 1.0 / 1.0`。而且 trainer 实现里也确实是按列表逐项计算再加权求和，不是口头上的“混合”。

更具体一点说：
- trainer 支持 `loss_type` 是一个 list，所以 `sigmoid / bco_pair / sft` 会被依次计算；
- 对每一项 loss，先求当前 batch 的 mean，再乘上对应 `loss_weight`，最后累加成 `total_loss`；
- `sft` 这项对应的是 `nll_loss`；
- `bco_pair` 不是直接走 TRL 默认实现，而是单独写了一版，用 `chosen/rejected logratio` 和一个 running mean `delta` 来算。

如果想把 `bco_pair` 说得更实现一点，可以说它大概是：

`L_bco = -logsigmoid(beta * chosen_logratio - delta) - logsigmoid(-(beta * rejected_logratio - delta))`

这里的 `delta` 是一个 running mean，用来做配对校正。

如果说的是在线 RL 这条线，那脚本里是单独的 `grpo`，用 QA reward 和 timestamp reward 去做 rollout 后的优化。这条线更强调 reward 驱动的相对比较，不是把 `sft loss` 直接显式写在同一个 `loss_type` 里。

所以面试里更安全的说法是：
- 我们有一条 MPO / offline RL 线，会把 preference loss 和 SFT anchor 一起做联合优化；
- 也有一条 online GRPO 线，核心是 rollout + reward。
- 如果面试官问“你们把 SFT 和 RL 怎么结合”，就先回答联合目标和 anchor 的作用，再回答为什么需要 reward 区分度。
- 如果面试官追到代码层，可以补一句：我们 trainer 里就是把多个 `loss_type` 展开后逐项算，再按 `loss_weights` 累加。

### 11B. 为什么 MPO 里要保留一个 SFT loss？

参考回答：

因为如果只做偏好项，模型很容易为了优化相对偏好而牺牲基础生成能力，出现格式漂移、描述覆盖率下降或者整体语言质量变差。把 SFT loss 作为 anchor 放进去，本质上是在告诉模型：你可以朝更高 reward 的方向走，但不要偏离原来已经学到的基本 caption 能力太远。所以联合 loss 的价值，不只是更稳，还能降低偏好学习把模型带坏的风险。

### 12. 你们怎么评估 caption 效果

参考回答：

我们不会只看一个总分，而是拆成几套独立信号。比如一套是 QA 准确率，覆盖 401 个视频、1349 道题，去看 caption 对视觉、音频和时间轴的理解是否正确；另一套是时间戳 IoU，分别评字幕和音频对话的定位效果。除此之外，我们还会保留 tricky case 和可视化报告，因为 caption 任务很容易出现“平均指标还行，但某类系统性错误一直在”的情况，所以自动评测和人工看 case 都很重要。

### 13. 你简历里写的 80%+ 指标具体是什么

参考回答：

这个一定要先澄清口径。更稳妥的回答方式是：简历里的 80%+ 更偏内部生产链路下的核心准确率或可用率口径，不等同于多镜头 benchmark 上的 QA raw accuracy。对外如果讲 benchmark，我会直接用统一 PE 当前更稳的说法，比如它在两版 QA 集上准确率稳定在 77%-78%，并且在 0319 QA 上高于 qwen3omni 当前训练版的 68.10% 和 Gemini 2.5 Pro 直出的 73.89%。这样口径更清楚，也更不容易被追问穿。

### 13A. 你们现在统一 PE 的结果大概是什么水平？

参考回答：

如果按现在统一 PE 的 benchmark 口径来讲，我会说它在两版 QA 集上都比较稳定。以我手上的结果为例，0324 版本在 0319 QA 上准确率是 78.36%，错误率 12.89%，不确定率 8.73%；在 0324 QA 上准确率是 77.68%。如果看 0319 QA 的横向对比，当前训练版 qwen3omni 是 68.10%，Gemini 2.5 Flash 直出是 63.60%，Gemini 2.5 Pro 直出是 73.89%，所以统一 PE 的结论不是“偶尔高一次”，而是相对直出方案和当前训练版都更稳。

### 14. 你们为什么会做 VLM + Omni 和 Omni 直出的对比

参考回答：

因为这两条路线各有优劣。VLM + Omni 的优点是画面描述可能更细、更稳定，但链路更长，整合成本更高。Omni 直出的优点是链路简单、理论上统一，但不一定在所有场景下都更强，尤其是多镜头和复杂视觉细节场景。所以必须通过实验验证，而不是先入为主选方案。

### 15. 你个人在这个项目里最难的问题是什么

参考回答：

我觉得最难的是怎么把多源信息组织成既适合训练、又适合评测、还能稳定生产的数据表示。因为视觉、音频、字幕、镜头切换这些信息天然是异构的，而且多镜头阶段和统一 PE 阶段服务的目标也不完全一样，所以 schema 不可能一版定终身。如果结构化表达和评测口径不清楚，后面训练和迭代都会变得很混乱。所以最难的其实不是单个模型，而是整条闭环的设计，以及 schema 如何跟着任务一起演进。

### 16. 你解决过一个具体的难问题吗？

参考回答：

如果讲一个更偏训练闭环的难问题，我会举 QA reward 的例子。早期 reward 设计最容易出的问题，不是规则写得不够多，而是同一次 rollout 里的答案没有区分度，结果经常接近全 0 或全 1。这样从训练视角看，相当于模型拿不到有效比较信号，GRPO 很难真正学到“哪种 caption 更好”。

后面我们的思路不是一味堆复杂规则，而是回到训练信号本身，去调整题目构造、评测方式和 reward 口径，让同一次 rollout 里的不同答案能被稳定拉开。这样 reward 不只是“能打分”，而是能提供有信息量的排序信号。这个点我觉得很关键，因为它直接决定 RL 是真的在学，还是只是在无效信号上震荡。

如果面试官更想听工程和多模态问题，我再补音频幻觉那个例子：模型容易看到字幕就误以为自己听到了对应内容。后面我们重构了音频链路，先做 qwen3-omni 语种检测，再用指定语言跑 Seed-ASR，把带时间戳的 ASR prior 注入后续音频理解流程，核心是先给模型一个可信语音锚点，减少音频侧的系统性幻觉。

## RL 追问：5 个 Killer Points

这部分不是背定义，而是面试官一旦往 RL 细节追，你至少要能讲出下面 5 个点。

### 1. reward 不是“能打分”就够了，关键是有没有区分度

推荐说法：

对 GRPO 来说，reward 最关键的不是形式多复杂，而是同一次 rollout 里的不同答案能不能被拉开。如果 QA reward 长期退化成全 0 或全 1，训练信号基本就失效了，因为模型看不到相对优劣。

### 2. reward 必须尽量贴近最终目标，而不是只优化代理指标

推荐说法：

caption 任务里最怕的是把模型训成只会投机某个自动指标。所以 reward 最好围绕 QA、时间定位、事实一致性这类可验证、又和真实质量强相关的信号来设计，而不是只看长度、格式或者某个单一分数。

### 3. RL 阶段一定要盯住基础生成能力有没有被搞坏

推荐说法：

RL 提升某些维度不代表整体更好。要持续看结构化输出有没有漂、描述覆盖率有没有掉、有没有出现新的模式坍缩，避免模型为了追 reward 牺牲基础 caption 能力。

### 4. 评测集和 reward 设计要联动，不然训练目标会飘

推荐说法：

如果评测关注的是 QA 和时间定位，但 reward 又主要奖别的东西，训练很容易跑偏。所以 reward、评测集、tricky case 和人工 review 最好形成一个闭环，而不是各做各的。

### 5. 多模态 RL 的难点不只是算法，还包括输入信号质量

推荐说法：

纯文本任务很多问题可以直接归到策略优化，但多模态 caption 里，输入侧本身就可能有噪声，比如音频幻觉、字幕污染、时间对齐错误。如果前面的结构化输入和评测口径没立住，后面的 RL 优化很容易建立在错误信号上。

## RL / MPO / GRPO 口语速答

这部分按“30 秒能答完”为目标来准备。重点不是把自己说成纯 RL 研究员，而是把你真正做过的训练闭环、loss 设计和 reward 经验讲清楚。

### 1. 如果面试官问：你们的 loss 怎么写？

推荐回答：

我们要分两条线说。offline MPO 这条线，本质上是联合 loss，不是单一目标。当前实现里是 `sigmoid + bco_pair + sft` 三项一起优化，trainer 会把每一项分别算出来，再按权重加权求和。这里 `sft` 对应的是 `nll loss`，更像一个 anchor，用来防止偏好优化把基础 caption 能力带偏。另一条是 online GRPO，这条线不是把 `sft loss` 显式写在同一个 `loss_type` 里，而是 rollout 后根据 reward 做相对优化。

### 2. 如果面试官问：为什么要保留 SFT loss？

推荐回答：

因为只做偏好项或者 reward 驱动项，模型很容易为了追某个目标把基础生成能力带偏，比如格式漂移、覆盖率下降或者语言质量变差。保留 SFT loss 的作用，本质上就是给模型一个 anchor，让它在往更高 reward 方向走的时候，不要偏离原来已经学到的稳定 caption 能力太远。

### 3. 如果面试官问：`bco_pair` 是什么？

推荐回答：

你可以把它理解成一个 pairwise 校正项。它不是只看 chosen 比 rejected 高不高，而是看它们相对 reference 的 logratio，再做一个 pairwise 的校正。我们实现里还加了 running mean `delta`，避免 reward 尺度太飘。你不用把公式背得特别死，但至少要知道它的作用是让 chosen / rejected 的相对差异更稳定。

### 4. 如果面试官问：你们为什么不用纯 GRPO 就够了？

推荐回答：

因为我们实际上有两条训练线，目标不完全一样。offline MPO 更适合把 preference 项和 SFT anchor 一起做联合优化，训练更稳；online GRPO 更强调 rollout + reward，适合继续围绕 QA 和 timestamp 这些可验证信号做相对优化。所以不是二选一，而是不同阶段和不同目标下各有适用性。

### 5. 如果面试官问：QA reward 最难的点是什么？

推荐回答：

最难的不是规则写不写得出来，而是训练信号有没有区分度。比如同一次 rollout 里，如果一批答案长期全 0 或全 1，reward 虽然存在，但信息量很低，GRPO 基本学不到什么。我们比较关注的是让同一次 rollout 里的不同答案能被稳定拉开，这样 reward 才真的是有效训练信号。

### 6. 如果面试官问：你们怎么判断 reward 设计得对不对？

推荐回答：

我不会只看 reward 定义本身，而会看三件事。第一，它是不是和最终目标一致，比如 QA、时间定位、事实一致性。第二，它在 rollout 后有没有区分度。第三，reward 提升之后，真实评测和 tricky case 有没有同步改善。如果这三件事对不上，reward 大概率就有问题。

### 7. 如果面试官问：你在 RL 这块具体做了什么？

推荐回答：

我更偏向训练闭环这一层，而不是底层 RL 框架开发。具体来说，我做得更多的是 reward 目标和评测维度怎么对齐、训练数据和样本构造怎么服务 RL、case 分析怎么回流到 reward 设计，以及训练结果怎么通过 QA / timestamp / tricky case 闭环验证。底层 trainer 框架我会看实现和参数，但不会说成是我从零写的。

## RL 基础追问题

这部分是为了防止面试官从工程细节切到基础概念。你不用讲得像研究员，但要保证逻辑顺。

### 1. 什么是 preference learning？

推荐回答：

就是模型不再只学“标准答案是什么”，而是学“多个候选答案里哪个更好”。所以它优化的是相对偏好，而不是单点监督标签。对 caption 这种任务来说，这种方式更适合表达“这个回答更好、更一致、更可信”，而不是只有绝对对错。

### 2. DPO 和 PPO 的大区别是什么？

推荐回答：

可以粗略理解成，PPO 更像经典在线 RL，要和策略更新、采样、优势估计这些东西一起配合；DPO 更偏离线偏好学习，把 preference 问题转成一个更直接的优化目标。面试里如果不是专门考 RL 理论，我一般不会展开算法推导，而是强调你们场景里为什么会选更稳定、可控的那条线。

### 3. GRPO 和 PPO 的区别你怎么理解？

推荐回答：

核心还是围绕 group relative comparison。也就是说它更强调同一组 rollout 样本里的相对优劣，而不是只盯一个绝对分数。对我们这种 caption 场景来说，这种相对比较更容易结合 QA reward、timestamp reward 这种可验证信号来做优化。

### 4. 为什么 reward 不能只看一个指标？

推荐回答：

因为单一指标很容易被投机。比如只看某个 QA 分数，模型可能学会迎合这类题；只看格式，模型可能输出特别保守。caption 真正关心的是事实性、时间定位、音画一致性、结构稳定性这些维度，所以 reward 最好是多维约束，不然会出现某一项很好、整体反而变差。

### 5. 为什么 RL 容易把模型带坏？

推荐回答：

因为 reward 往往只是目标的一个近似，如果优化得太猛，模型可能会学到“怎么拿高分”，而不是“怎么真正变好”。这就会表现成格式漂移、回答变模板化、覆盖率下降、或者某些 tricky case 全面退化。所以 RL 阶段必须一直盯基础能力有没有被破坏。

### 6. 什么叫 reward hacking？

推荐回答：

就是模型学会钻 reward 的空子，而不是学会真正的目标能力。比如它输出一些模板化但讨巧的内容，自动分数看起来涨了，但人看或者 tricky case 并没有更好。避免 reward hacking 的方法，通常就是多维评测、人工回看、tricky case 回归，以及保证 reward 和真实目标尽量一致。

### 7. 什么是 reference model？为什么需要它？

推荐回答：

你可以把 reference model 理解成一个相对稳定的参照系。很多 preference / DPO 类方法里，chosen 和 rejected 不只是看当前模型本身的差异，还会和 reference 的 logprob 做比较。这样做的好处是让优化更有约束，不至于模型随便漂。

### 8. 什么是 chosen / rejected？

推荐回答：

就是一对候选答案里的“较优答案”和“较差答案”。在 caption 任务里，它不一定是完全正确和完全错误，更常见的是一个更符合事实、更完整、更一致，另一个相对差一些。偏好学习的核心就是让模型更倾向 chosen、远离 rejected。

### 9. 为什么多模态 caption 的 RL 比纯文本更难？

推荐回答：

因为它除了输出端的问题，还有输入端的问题。纯文本任务主要担心生成策略和 reward 定义；多模态 caption 还要担心视觉、音频、字幕、时间信息本身是否可靠。如果输入信号有噪声，reward 再漂亮也可能是在优化一个错的目标。

### 10. 如果面试官继续追公式，但你不想硬扛怎么办？

推荐回答：

更安全的方式不是乱背公式，而是先把训练目标、相对比较关系、anchor 的作用和实现路径讲清楚。比如你可以说：我们这条 MPO 线在 trainer 里是按 `loss_type` 列表逐项计算再加权求和，`sft` 是 `nll loss`，`bco_pair` 是 pairwise 校正项，核心作用是稳住基础生成能力并拉开 preference。这样通常已经够用了。如果对方还要往理论推导深挖，再诚实说自己更偏训练落地和闭环，而不是 RL 理论研究。

## PE 项目设计 rationale

### 1. 为什么不是直接让模型一步生成 caption？

参考回答：

因为一步生成虽然链路最短，但问题也最明显。第一是可控性差，很难知道错误到底出在视觉、音频还是字幕理解。第二是可解释性差，出了问题不容易定位和修复。第三是大规模生产时稳定性不够，很多中间结果没法复用。所以我们后来把它拆成多阶段 pipeline，不是为了让流程复杂，而是为了让每一阶段都能单独优化、缓存和回归。

如果面试官继续追问“有没有证据说明 PE 确实比直出强”，可以补一句：以 0319 QA 为例，Gemini 2.5 Pro 直出是 73.89%，当前统一 PE 版本是 78.36%，当前训练版 qwen3omni 是 68.10%，所以 PE 不是纯工程堆砌，而是确实把评测结果拉上来了。

### 2. preprocess 阶段为什么重要？

参考回答：

preprocess 的作用是先把视频里的基础信息尽量提纯出来，比如镜头切分、语种检测、ASR、音频信息、OCR、关键帧和 prescan 等。这些信息本身不一定是最终结果，但它们会给主 caption 提供比较稳定的锚点。尤其在多镜头和带音频场景里，如果没有这些前置信号，模型很容易在后面一步里把视觉、语音和字幕混在一起。

### 3. 为什么要把音频单独拉出一条链路？

参考回答：

因为音频问题和视觉问题很不一样。我们实际遇到的一个长期问题就是模型会被字幕干扰，看到字就觉得听到了对应内容，这会直接导致音频幻觉。所以音频不能只作为主 caption 里的一个附属输入，而要单独做 qwen3-omni 语种检测、指定语言 Seed-ASR、audio caption，再把相对可信的结果注入主流程。这样做不是为了增加模块，而是为了降低系统性错误，并且它能在多语言视频里明显降低 ASR 错误带来的连锁问题。

### 4. 为什么要做 QA revise，而不是只靠主 caption？

参考回答：

因为主 caption 的问题往往不是完全不会生成，而是局部事实错、顺序错、细节漏或者不同模态之间有冲突。QA revise 的价值就是把“生成”变成“生成 + 核验 + 修订”。让模型先针对 caption 生成高价值问题，再回到视频里核验，最后只修正真正冲突的部分。这样做的好处是改动范围更小，也更适合回归验证。

### 5. 为什么需要做 schema 演进？

参考回答：

如果单视频、多镜头、带音频和不带音频的数据格式都完全割裂，后面的训练和评测会非常碎；但如果强行要求所有阶段永远共用一套完全相同的 schema，也不现实。更准确的说法是要做 schema 演进。早期多镜头 schema 更偏评测和训练可用性，后来的统一 PE schema 更偏多源融合、主 caption、修订和最终数据生产。我的工作重点是让这两类结构化表达在各自阶段都足够清晰、可验证，并且整体上服务同一条训练闭环。

### 6. 为什么要保留 tricky case 和可视化分析？

参考回答：

因为很多多模态问题不是只靠一个总指标就能看出来的。尤其 caption 任务里，经常会出现“平均分没掉，但某类系统性错误一直存在”的情况。tricky case 和可视化分析的价值，就是把这些高风险错误显式沉淀下来，避免下一个版本又回退。对工业系统来说，这种回归机制非常关键。

### 7. 怎么把这套 pipeline 讲得像“方法论”而不是“工程堆砌”？

参考回答：

我一般会把它概括成三层：第一层是信息提纯，也就是 preprocess；第二层是结构化表达，也就是主 caption 和 schema 演进；第三层是质量闭环，也就是 QA revise、评测和人工修正。这样讲就不是在堆模块，而是在讲一套从数据到质量控制的完整方法。

### 17. 如果让你从 0 到 1 搭一个视频 caption 系统，你会怎么做

参考回答：

我会先定任务边界和第一版结构化表示，再定评测口径，然后做一个可跑通的 baseline。之后再补数据生产链路，把预处理、主 caption、修订和评测串起来。有了稳定 baseline 以后，再根据训练目标和评测反馈去迭代表达方式，而不是假设第一版 schema 永远不变。我的经验是先把闭环做完整，再推动 schema 和训练目标一起演进，比一开始就追单点最优更重要。

#### 参考回答

### 1. Transformer 的 self-attention 为什么有效？

参考回答：

self-attention 的核心优势是它能让每个 token 直接和所有其他 token 建立关系，不像 RNN 那样必须一步一步传递信息，所以更容易建模长距离依赖。而且它天然适合并行计算，训练效率也更高。对多模态任务来说，这种全局建模能力也更适合做跨区域、跨时间的信息对齐。

### 2. KV cache 是什么，为什么能加速推理？

参考回答：

自回归推理时，历史 token 的 K 和 V 一旦算出来就不会变，所以没必要每生成一个新 token 都把整段历史重新算一遍。KV cache 就是把历史 token 的 K/V 保存下来，下一步只算当前 token，再和缓存的 K/V 交互。它本质上是用显存换延迟，长上下文里收益尤其明显。

### 3. 多头注意力相比单头注意力的意义是什么？

参考回答：

多头注意力可以理解成让模型在不同子空间里同时看关系。不同 head 往往会关注不同模式，比如有的看局部动作，有的看主体关系，有的看时序或位置。相比单头，它表达能力更强，也更容易学到互补信息。

### 4. RoPE 大概解决了什么问题？

参考回答：

RoPE 主要解决的是位置编码问题。因为 self-attention 本身对顺序不敏感，所以必须显式注入位置信息。RoPE 的好处是它把位置信息放进 Q/K 的旋转里，让 attention 更自然地建模相对位置关系，在长文本和长视频场景里都比较常见。

### 5. SFT、DPO、PPO、GRPO 的区别是什么？

参考回答：

SFT 是监督微调，本质上是教模型按目标格式和行为去生成。DPO 更偏离线偏好学习，学习的是候选答案之间谁更好。PPO 是更经典的在线 RL 方法，涉及策略更新和优势估计。GRPO 可以理解成更强调同组 rollout 样本之间的相对比较，比较适合结合可验证 reward 去做优化。面试里不一定要背算法细节，但要知道它们分别服务什么训练目标。

### 6. RLHF 里 reward model 和 rule-based reward 的差别是什么？

参考回答：

reward model 更像是学出来的评分器，优点是表达能力强，但训练成本高，也更容易带入偏差。rule-based reward 更像人工定义的可验证规则，比如 QA 对错、时间定位是否匹配，优点是可解释、稳定、好排查。你们这个场景更适合先围绕可验证信号做 rule-based reward，因为 caption 的很多关键维度本身就可以通过 QA 和 timestamp 这类方式自动评估。

### 7. 为什么长序列训练更难？主要难在哪些资源瓶颈？

参考回答：

长序列训练最直接的瓶颈是 attention 的算力和显存开销，因为复杂度会随序列长度快速上升。除此之外，多模态场景还会带来更多 token，比如视频帧、OCR、音频信息一起进来，序列更长、batch 更小、训练更不稳定。所以长序列训练通常要结合并行、重计算、FlashAttention、token 控制这些手段一起做。

### 8. 多模态模型里视觉特征通常怎么接到 LLM 上？

参考回答：

最常见的做法是先用视觉编码器把图像或视频变成视觉 token，再通过 projector 或对齐模块映射到 LLM 可以接收的表示空间，然后和文本 token 一起喂给模型。不同架构差别主要在 projector 怎么做、视觉 token 怎么压缩、以及是否对时序做额外建模。

### 9. 视频输入相比图像输入，多了哪些建模难点？

参考回答：

视频除了空间理解，还要处理时间维度，所以难点会多很多。比如事件顺序、镜头切换、长程依赖、对白和画面的对应关系、字幕和音频是否一致，这些都是图像任务里没有或没那么强的问题。视频一旦再叠加音频和 OCR，就会变成典型的多源异构问题。

### 10. 为什么多镜头场景比单镜头更难？

参考回答：

因为多镜头不是简单多几张图，而是要理解镜头间的连续性和切换关系。主体可能跨镜头延续，语音和字幕也可能跨镜头延续，局部看对了不代表整体时序没错。所以多镜头 caption 会更依赖结构化表示、时间信息和评测闭环。

4. 自动评测和人工评测各自适合什么场景？

6. 你会怎么定义 caption 的“事实性”？
7. 你会怎么定义“音画一致性”？
8. 如果一个方案指标更高，但人眼看更差，你会怎么处理？

#### 参考回答

### 1. 高质量训练数据最重要的几个维度是什么？

参考回答：

我会优先看四个维度：准确性、完整性、一致性和分布匹配。准确性是内容不能错，完整性是关键信息不能漏，一致性是视觉、音频、字幕、时间轴之间不能互相打架，分布匹配是训练数据得和目标任务真的相关。对 caption 来说，这四个维度通常比单纯数据量更重要。

### 2. 训练数据脏会造成哪些典型问题？

参考回答：

最常见的结果就是模型学到错误映射，比如音频幻觉、字幕污染、时间顺序错误、格式漂移，或者输出特别模板化。更麻烦的是这些问题有时候不会立刻反映在一个总指标上，而是会在 tricky case 上长期存在，所以脏数据对生成任务尤其危险。

### 3. 什么叫数据分布不一致？为什么会影响效果？

参考回答：

数据分布不一致就是训练时看到的样本和推理时真实会遇到的样本差别太大，比如训练里几乎没有多语言音频、多镜头切换或者复杂 OCR，线上却很多。这样模型虽然在训练集上看起来正常，但一到真实场景就会暴露短板。对多模态任务来说，分布偏差常常比模型结构本身更致命。

### 4. 自动评测和人工评测各自适合什么场景？

参考回答：

自动评测适合大规模、稳定、可回归的场景，尤其是 QA、时间定位这类可以规则化的指标。人工评测更适合发现系统性错误、理解 case 质量，以及检验自动指标没覆盖到的问题。最好的方式不是二选一，而是自动评测做主干，人工 case review 做兜底。

### 5. 为什么很多生成任务不能只看单一指标？

参考回答：

因为生成任务很容易出现“一个指标涨了，但真实质量没变好”的情况。比如格式更规整了，不代表事实性更强；QA 分数涨了，也不一定说明所有 case 都更自然。所以生成任务最好拆成多个维度去看，尤其要防止模型学会投机某个单一指标。

### 6. 你会怎么定义 caption 的“事实性”？

参考回答：

我会把事实性理解成生成内容能不能被视频本身支持。也就是说主体、动作、对白、环境音、字幕文字和时间顺序都不能胡编。对 caption 来说，事实性不只是“有没有明显错”，还包括有没有把不同模态之间本来不一致的信息强行拼在一起。

### 7. 你会怎么定义“音画一致性”？

参考回答：

音画一致性不是简单把画面和音频都写一遍，而是视觉事件、音频事件、字幕文字和时间轴要彼此对齐。比如谁在说话、说的内容、画面里的动作、字幕显示的文本、发生的先后顺序，都不能互相冲突。这个维度对训练数据是否可用非常关键。

### 8. 如果一个方案指标更高，但人眼看更差，你会怎么处理？

参考回答：

我不会直接相信指标，而是先拆口径。要看提升的是哪个指标、是不是 reward hacking、是不是某类 case 被牺牲掉了、人工抽样是不是出现了系统性退化。如果自动分数更高但真实可视化结果更差，我会优先回到 case 和评测定义，而不是继续沿着这个方向优化。

#### 参考回答

### 1. 多阶段 pipeline 为什么比单阶段更可控？

参考回答：

因为它把一个复杂问题拆成几个可解释的小问题。这样每一阶段都能单独看输入、输出、缓存和失败样例，问题更容易定位，也更适合回归验证。对工业系统来说，可控、可解释、可重跑通常比“链路最短”更重要。

### 2. 并发处理时最常见的问题有哪些？

参考回答：

最常见的是资源争抢、下游 API 被打满、结果落盘不完整、失败重试把数据打乱，以及缓存和幂等做不好。多模态任务里还会叠加大文件下载、视频解码、音频处理这些额外开销，所以并发设计必须和限流、重试、落盘策略一起考虑。

### 3. 缓存复用为什么重要？什么情况下容易出错？

参考回答：

缓存复用的重要性在于节省重复计算，尤其视频预处理和模型推理成本很高的时候，没缓存会非常浪费。容易出错的地方主要是版本不一致、字段依赖变化、缓存命中条件写得太粗，最后导致“命中了旧结果但你自己没发现”。所以缓存一定要和阶段、配置、版本一起管理。

### 4. 你如何设计一个可回归的评测闭环？

参考回答：

我会先明确核心指标，再保留一组稳定的测试集和 tricky case 集。每次版本更新都同时看自动评测、关键 case 可视化和必要的人审结果，避免只靠一个表格判断好坏。最关键的是评测结果要能稳定复现，并且能回溯到具体样例和具体阶段。

### 5. 大规模数据生产里最容易出现哪些工程问题？

参考回答：

最常见的是数据源不稳定、字段缺失、并发失败、结果重复写入、缓存污染、视频路径失效和版本口径不一致。更深一层的问题是，很多工程 bug 最后会伪装成“模型效果变差”，所以大规模数据生产必须把日志、幂等、版本管理和回归机制一起做好。

## Coding 题

这些题不一定真的按 LeetCode 形式出，但很容易被拿来考察工程基本功。你至少要能白板/口述出思路。

### 字符串 / 解析类

1. 给一个字符串，统计其中每种 XML 标签出现次数，比如 `<speech>...</speech>`、`<text>...</text>`。
2. 给一段结构化 caption 文本，提取所有带时间信息的事件。
3. 判断一个字符串是否是合法 JSON，或者如何做一个健壮的 JSON 解析重试。
4. 实现一个简单模板替换器，把 `__ASR__`、`__OCR__` 这类占位符替换掉。

### 区间 / 时间戳类

1. 给定一组时间区间，合并重叠区间。
2. 计算两个时间区间的 IoU。
3. 给一个事件序列，判断是否按时间递增。
4. 给字幕片段和 ASR 片段，做一个简单时间对齐。

### 数据处理类

1. 从 JSONL 文件里读取数据，做过滤、清洗、重写输出。
2. 对超大 JSONL 文件做去重，你会怎么设计？
3. 给多源字段做 join，比如把 caption 结果和元数据按 `video_id` 合并。
4. 实现一个支持断点续跑的批处理脚本。

### 并发 / 系统设计类

1. 设计一个多线程或多进程视频处理流水线。
2. 如果一个批处理任务有 10 万条数据，怎么做失败重试和结果落盘？
3. 如何限制并发，避免把下游 API 打挂？
4. 多个阶段有依赖关系时，怎么设计缓存和重跑机制？
