# 面试问答稿

## 从易到难

### 1. 你先简单介绍一下自己

参考回答：

我这几年主要在腾讯做视频理解和多模态 caption 相关工作，核心方向是视频 caption、带音频的 omni caption，以及多镜头 caption 的方案研发。工作内容不是只做单点模型，而是从 caption 格式设计、数据构造、训练到评测闭环一起推进。更早之前我在 Shopee 和小米做推荐和广告算法，所以也有比较完整的工业落地和效果迭代经验。现在我主要希望找偏多模态 LLM 落地、视频理解和数据训练闭环相关的岗位。

### 2. 你最近最核心的项目是什么

参考回答：

最近最核心的是视频 caption / omni caption / 多镜头 caption 方案研发。这个项目经历了几个阶段：最开始是纯视频、没有音频的 caption；后来扩展到带音频的多镜头 caption；再往后收敛成统一的 caption / omni caption 方案。我的工作重点在于把这条链路做成一个可持续迭代的系统，包括 schema、数据生产、SFT、GRPO 和评测闭环。

### 3. 你在这个项目里主要负责什么

参考回答：

我主要负责四块。第一是方案演进和选型，比如不同阶段怎么在纯视觉、带音频、多镜头和 omni 之间取舍。第二是统一 caption schema 设计，让结果同时适合数据生产、训练和评测。第三是训练数据和评测集建设，把从预处理到主 caption 再到修订的流程打通。第四是训练和效果闭环，包括 SFT、GRPO，以及围绕 QA 和时间定位的评测和优化。

### 4. 为什么要从纯视频 caption 逐步演进到 omni caption

参考回答：

一开始做纯视频 caption，主要是因为路径最短，先解决画面理解和基础描述能力。但后面很快发现，很多视频任务里音频是关键增量信息，比如对白、旁白、环境音、歌词，都会直接影响事实性和完整性。尤其到了多镜头场景，镜头切换、语音、字幕和画面之间的关系更复杂，如果只看画面会丢掉很多关键信息，所以后面才逐步演进到统一的 omni caption 方案。

### 5. 你说的“音画一致 caption”具体是什么意思

参考回答：

我理解的音画一致，不是简单把画面和音频都描述一遍，而是让最终 caption 在视觉事件、音频事件、字幕文字和时间顺序上保持一致。比如画面里是谁在做什么、对应的台词是谁说的、字幕和语音是否一致、事件先后顺序是否正确，这些都要能对齐。否则模型虽然生成了很多内容，但不一定适合拿来训练。

### 6. 统一 caption schema 的价值是什么

参考回答：

统一 schema 的核心价值是把不同阶段和不同类型的数据拉到同一个接口上。因为我们既有单视频 caption，也有多镜头 caption，还有带音频的 omni caption。如果每一类数据格式都不一样，后面的数据生产、训练和评测都会很碎。统一 schema 之后，视觉、音频、字幕 OCR、时间信息都有清晰字段，数据更容易构造，训练目标更稳定，评测也更容易做。

### 7. 你们为什么要做多阶段 pipeline

参考回答：

多阶段 pipeline 主要是为了把复杂问题拆开。直接让一个模型端到端做所有事情，虽然看起来简单，但可解释性、稳定性和可控性都差。拆成预处理、主 caption、QA 修订这些阶段后，不同信息源可以独立优化，失败样例也更容易定位问题。同时这样更适合做缓存复用和批量生产，工程上更稳。

### 8. 你提到 QA 自修正流程，这个是怎么工作的

参考回答：

核心思路是让模型先基于 caption 自己提出高价值验证问题，然后再基于视频回答这些问题，最后把回答结果和原 caption 做对比，只修正真正冲突的部分。这样做的价值是把“生成”变成“生成 + 核验 + 修订”的闭环。对 caption 这种任务来说，很多问题不是不会写，而是容易出现局部幻觉、时间顺序错误或者细节遗漏，QA 自修正对这些问题比较有效。

### 9. 训练数据是怎么构造的

参考回答：

训练数据不是单纯把模型输出直接拿来训，而是有一整套生产链路。包括视频预处理、镜头信息、ASR、音频描述、字幕或 OCR 信息的整合，再到结构化 caption 生成、过滤、修订和最终入库。我主要做的是把这些环节串起来，并保证数据格式统一、质量可控，能同时服务 SFT、RL 和评测。

### 10. SFT 和 GRPO 各自解决什么问题

参考回答：

SFT 主要解决的是“模型能不能稳定按我们想要的格式和粒度生成 caption”。也就是说先把基本能力学出来，包括结构化输出、描述完整性和不同信息源的整合。GRPO 解决的是进一步对齐问题，比如事实性、时间定位、时序一致性和音画对齐。这些能力单靠 SFT 往往不够稳定，所以会再结合 reward 去优化。

### 11. 你们的 reward 大概怎么设计

参考回答：

总体上是围绕结果可验证的维度设计，比如 QA 相关的内容理解、时间定位，以及生成结果和视频事实的一致性。原则不是把 reward 设计得很花，而是让模型输出能被稳定自动打分，并且这个分数和我们真正关心的效果相关。这样 RL 才有意义，否则只是在优化一个和目标脱节的代理指标。

### 12. 你们怎么评估 caption 效果

参考回答：

我们不会只看一个总分，而是拆成几个关键维度，比如内容理解、时间定位、音画一致性，以及一些典型幻觉 case 的表现。一方面有自动化评测流程，另一方面也会看典型 case 的可视化结果。因为 caption 任务很容易出现“指标看起来没问题，但人眼一看不对”的情况，所以自动评测和人工看 case 都很重要。

### 13. 你简历里写的 80%+ 指标具体是什么

参考回答：

这个一定要按你真实口径回答。推荐回答方式是先说这是内部评测体系下的核心准确率指标，再补一句它覆盖的评测维度。重点不是绝对数值，而是说明它来自稳定的内部评测流程，而不是单次主观观察。

### 14. 你们为什么会做 VLM + Omni 和 Omni 直出的对比

参考回答：

因为这两条路线各有优劣。VLM + Omni 的优点是画面描述可能更细、更稳定，但链路更长，整合成本更高。Omni 直出的优点是链路简单、理论上统一，但不一定在所有场景下都更强，尤其是多镜头和复杂视觉细节场景。所以必须通过实验验证，而不是先入为主选方案。

### 15. 你个人在这个项目里最难的问题是什么

参考回答：

我觉得最难的是怎么把多源信息统一成一个既适合训练、又适合评测、还能稳定生产的数据表示。因为视觉、音频、字幕、镜头切换这些信息天然是异构的，如果 schema 不清楚、评测口径不清楚，后面训练和迭代都会变得很混乱。所以最难的其实不是单个模型，而是整条闭环的设计。

### 16. 你解决过一个具体的难问题吗？

参考回答：

有一个很典型的问题是音频幻觉。我们用多模态模型做视频打标时，经常会出现“模型看到字幕，就以为自己听到了对应内容”的情况，也就是视觉信息反过来污染了音频理解。这种问题以前一直比较难解决，因为单纯改 prompt 效果有限。

后来我们不是只在提示词上修，而是重构了音频链路：先做语种检测，再做更可靠的 ASR，把 ASR prior 注入到后续音频理解流程里，让模型先拿到可信的语音锚点，再去做音频事件描述。这个改动的核心是先约束模型，减少它在音频侧的系统性幻觉。这个链路上线后，对多语言场景和音画一致性都有明显帮助。

## PE 项目设计 rationale

### 1. 为什么不是直接让模型一步生成 caption？

参考回答：

因为一步生成虽然链路最短，但问题也最明显。第一是可控性差，很难知道错误到底出在视觉、音频还是字幕理解。第二是可解释性差，出了问题不容易定位和修复。第三是大规模生产时稳定性不够，很多中间结果没法复用。所以我们后来把它拆成多阶段 pipeline，不是为了让流程复杂，而是为了让每一阶段都能单独优化、缓存和回归。

### 2. preprocess 阶段为什么重要？

参考回答：

preprocess 的作用是先把视频里的基础信息尽量提纯出来，比如镜头切分、ASR、音频信息、OCR、关键帧等。这些信息本身不一定是最终结果，但它们会给主 caption 提供比较稳定的锚点。尤其在多镜头和带音频场景里，如果没有这些前置信号，模型很容易在后面一步里把视觉、语音和字幕混在一起。

### 3. 为什么要把音频单独拉出一条链路？

参考回答：

因为音频问题和视觉问题很不一样。我们实际遇到的一个长期问题就是模型会被字幕干扰，看到字就觉得听到了对应内容，这会直接导致音频幻觉。所以音频不能只作为主 caption 里的一个附属输入，而要单独做语种检测、ASR、音频分析，再把相对可信的结果注入主流程。这样做不是为了增加模块，而是为了降低系统性错误。

### 4. 为什么要做 QA revise，而不是只靠主 caption？

参考回答：

因为主 caption 的问题往往不是完全不会生成，而是局部事实错、顺序错、细节漏或者不同模态之间有冲突。QA revise 的价值就是把“生成”变成“生成 + 核验 + 修订”。让模型先针对 caption 生成高价值问题，再回到视频里核验，最后只修正真正冲突的部分。这样做的好处是改动范围更小，也更适合回归验证。

### 5. 为什么需要统一 schema？

参考回答：

如果单视频、多镜头、带音频和不带音频的数据格式都不一样，后面的训练和评测会非常碎。统一 schema 的意义在于，把视觉事件、音频事件、字幕/OCR、时间信息放到一个稳定接口里。这样数据生产、模型训练、自动评测和人工修正都可以围绕同一套表示做迭代，整个系统才有办法持续优化。

### 6. 为什么要保留 tricky case 和可视化分析？

参考回答：

因为很多多模态问题不是只靠一个总指标就能看出来的。尤其 caption 任务里，经常会出现“平均分没掉，但某类系统性错误一直存在”的情况。tricky case 和可视化分析的价值，就是把这些高风险错误显式沉淀下来，避免下一个版本又回退。对工业系统来说，这种回归机制非常关键。

### 7. 怎么把这套 pipeline 讲得像“方法论”而不是“工程堆砌”？

参考回答：

我一般会把它概括成三层：第一层是信息提纯，也就是 preprocess；第二层是统一表达，也就是主 caption 和 schema；第三层是质量闭环，也就是 QA revise、评测和人工修正。这样讲就不是在堆模块，而是在讲一套从数据到质量控制的完整方法。

### 17. 如果让你从 0 到 1 搭一个视频 caption 系统，你会怎么做

参考回答：

我会先定任务边界和 schema，再定评测口径，然后做一个可跑通的 baseline。之后再补数据生产链路，把预处理、主 caption、修订和评测串起来。有了稳定 baseline 以后，再分阶段做 SFT、自动修正和 RL，对每一步都保留可解释的中间结果和回归 case。我的经验是先把闭环做完整，比一开始就追单点最优更重要。

## Coding 题

这些题不一定真的按 LeetCode 形式出，但很容易被拿来考察工程基本功。你至少要能白板/口述出思路。

### 字符串 / 解析类

1. 给一个字符串，统计其中每种 XML 标签出现次数，比如 `<speech>...</speech>`、`<text>...</text>`。
2. 给一段结构化 caption 文本，提取所有带时间信息的事件。
3. 判断一个字符串是否是合法 JSON，或者如何做一个健壮的 JSON 解析重试。
4. 实现一个简单模板替换器，把 `__ASR__`、`__OCR__` 这类占位符替换掉。

### 区间 / 时间戳类

1. 给定一组时间区间，合并重叠区间。
2. 计算两个时间区间的 IoU。
3. 给一个事件序列，判断是否按时间递增。
4. 给字幕片段和 ASR 片段，做一个简单时间对齐。

### 数据处理类

1. 从 JSONL 文件里读取数据，做过滤、清洗、重写输出。
2. 对超大 JSONL 文件做去重，你会怎么设计？
3. 给多源字段做 join，比如把 caption 结果和元数据按 `video_id` 合并。
4. 实现一个支持断点续跑的批处理脚本。

### 并发 / 系统设计类

1. 设计一个多线程或多进程视频处理流水线。
2. 如果一个批处理任务有 10 万条数据，怎么做失败重试和结果落盘？
3. 如何限制并发，避免把下游 API 打挂？
4. 多个阶段有依赖关系时，怎么设计缓存和重跑机制？

## 薪资与 Level 说法

### 1. 如果目标是 130 万左右

这个 level 更像“强资深算法工程师 / 资深研究工程师”。

面试里要强调：

- 我这几年持续在做多模态视频 caption，不是临时转方向
- 我做的是完整闭环，不只是单点 prompt 或模型调用
- 我能独立负责数据、训练、评测的一整段工作

推荐说法：

我比较适合承担一个明确方向的核心工作，比如视频 caption 或多模态数据训练闭环。我过去做的不是单点模块，而是从方案迭代、数据构造到训练和评测一起推进，所以我认为自己已经是资深工程师水平，能够独立扛住一个方向的落地。

### 2. 如果目标是 150 万左右

这个 level 开始需要明显的 technical leadership 信号。

面试里要强调：

- 你不只是做模块，而是在推动一条路线演进
- 你对技术选型有清晰判断
- 你能连接数据、训练、评测和人工修正流程
- 你能定义规则、标准和闭环

推荐说法：

我不是传统的 people manager，但我的角色也不只是执行。我更偏技术主导型，一方面会亲自做关键设计和落地，另一方面会推动数据生产、模型训练、自动评测和人工修正这些环节协同，保证方案能从实验验证走到规模化生产。我认为自己比较适合承担更大 scope 的技术责任。

### 3. 如果目标是 180 万以上

这个 level 通常要让面试官相信你已经接近“专家岗 / 核心骨干”。

面试里要强调：

- 你做的是团队级、方向级的方法和标准
- 你的工作影响的是后续多个环节，而不是单次实验
- 你能抽象出可复用的方法论
- 你有能力在复杂不确定问题里做技术取舍

推荐说法：

我过去做的不只是某一个 caption 模块，而是围绕视频 caption / omni caption 这条方向，逐步推动统一 schema、统一评测口径和统一数据生产方式的建立。我的优势不在于单点模型创新，而在于能把一个复杂方向从方案验证推进到稳定生产，并沉淀成可持续迭代的系统。如果岗位需要的是这类技术负责人型角色，我认为我是匹配的。

## 初聊时怎么判断机会值不值得继续面

### 核心原则

初聊最重要的不是先证明自己，而是先判断对方有没有能力正确理解和定价你的价值。

### 1. 他们到底在招什么人

你要尽快判断这个岗位更偏：

- 模型本体
- 数据 / 训练 / 评测闭环
- 业务应用
- 杂糅型通才

适合你的，是偏视频 / 音频多模态、数据训练评测闭环、research engineer 或技术主导型落地角色。

可问的问题：

- 这个岗位更偏模型本体、数据训练评测，还是业务应用？
- 这个岗位进来后最核心要解决的问题是什么？

### 2. 这个组认不认你的工作是主线价值

你要判断他们是否认可：

- caption / schema / 数据质量
- 训练与评测闭环
- 人工修正与回流

这些是主线关键能力，而不是 support。

可问的问题：

- 数据、评测和训练闭环在这个方向里是核心能力，还是偏 support 的角色？
- 这个岗位在团队里的核心价值更偏哪一段？

### 3. 岗位给不给更大的 scope

你现在已经不是只看“有没有岗位”，而是看“是不是更大一级的问题”。

可问的问题：

- 这个岗位更多是承接已有方案，还是需要自己推进方向演进？
- 岗位 scope 更偏单点优化，还是需要统筹数据、训练、评测这些环节？

### 4. 具体 leader 懂不懂你的价值

同一家公司，不同 leader 差异会非常大。

你要观察：

- 他追问的是方法论和系统设计，还是只盯某个 model trick
- 他会不会主动问数据、评测、质量闭环
- 他是否理解多模态系统里上游关键能力的价值

### 5. 团队有没有资源和耐心做这件事

复杂多模态系统需要：

- 数据
- 算力
- 评测体系
- 人工修正 / 标注能力
- 一定试错空间

可问的问题：

- 这个方向现在是探索期，还是已经有稳定资源和明确目标？
- 团队在数据、评测、训练资源上大概是什么配置？
- 这个岗位对应的核心成果，会更看实验指标、业务落地，还是系统建设？

### 6. package 和 level 有没有真实升级空间

你要判断：

- 对方把你看成资深工程师，还是专家候选
- 预算在哪个档位
- scope 是否真的比现在更大

可问的问题：

- 这个岗位在团队里大概是什么 level 预期？
- 这个岗位对应的 package 区间，大概落在哪个范围？
- 如果人选特别匹配，scope 和 level 是否有上调空间？

## 初聊后的判断标准

### 值得继续面的信号

- 对方能听懂你在讲的是方向级闭环，而不是只把你当成“会调 prompt 的人”
- 岗位 scope 明显大于单模块开发，确实需要方案判断和跨环节协同
- leader 会主动聊数据、训练、评测、质量回流，而不只是问某个模型 trick
- 团队有真实资源做事，不是只有模糊方向，没有数据、算力和评测条件
- level 和 package 至少有实质升级空间

### 应该谨慎的信号

- 对方只想找一个执行已有流程的人
- 岗位实际是杂活型通才，边界非常散，但没有明确主线
- leader 不理解数据和评测闭环的价值，只把这部分看成 support
- 团队还在非常早期，但又希望短时间内立刻出大结果
- 薪资和职责都接近你当前状态，没有明显升级

## 最后建议

### 面试准备顺序

1. 先把 1 分钟自我介绍和核心项目讲顺。
2. 再把 5 个必须准备清楚的问题练到不需要思考。
3. 然后补技术基础题和 coding 题，保证不会在基础环节失分。
4. 最后准备谈薪和机会判断话术，避免到了后期临场被带节奏。
